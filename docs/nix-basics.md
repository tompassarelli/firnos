# NixOS Basics: Store, Symlinks, and Configuration

## How NixOS Works: The /nix/store

On traditional Linux, you edit config files directly. On NixOS, config files are **read-only symlinks** to `/nix/store`:

```bash
ls -la /etc/nginx/nginx.conf
# lrwxrwxrwx  /etc/nginx/nginx.conf -> /nix/store/abc123-nginx.conf
```

The `/nix/store` is:
- **Immutable**: Files cannot be modified after creation
- **Content-addressed**: Each path includes a hash of its inputs
- **Shared**: Multiple configurations reuse identical files

### Configuration Flow

```
1. You write:       ~/code/nixos-config/*.nix
                              ↓  (nix evaluates)
2. Nix generates:   /nix/store/hash-nginx.conf
                              ↓  (creates symlink)
3. System uses:     /etc/nginx/nginx.conf -> /nix/store/hash-nginx.conf
```

### What `nixos-rebuild switch` Does

1. **Evaluates** your `.nix` files
2. **Builds** any packages needed
3. **Generates** config files → `/nix/store`
4. **Creates symlinks** from `/etc` → `/nix/store`
5. **Activates** new configuration (restarts services, switches units)
6. **Updates boot entry**

This makes your system **declarative** (state is in `.nix` files), **reproducible** (same inputs = same outputs), **atomic** (switch fails cleanly), and **rollbackable** (old configs stay in the store).

## mkOutOfStoreSymlink: Live-Editing Configs

The /nix/store approach means every config change requires a rebuild. For configs you tweak frequently (keybinds, editor settings), this is slow.

**Solution**: symlink directly to your dotfiles, bypassing the store:

```nix
# modules/niri/niri.nix
home-manager.users.${username} = { config, ... }: {
  xdg.configFile."niri/config.kdl".source =
    config.lib.file.mkOutOfStoreSymlink
      "${config.home.homeDirectory}/code/nixos-config/dotfiles/niri/config.kdl";
};
```

**Normal**: source → `/nix/store/hash-config` → `~/.config/niri/config.kdl` (read-only, requires rebuild)

**Out-of-store**: source → `~/.config/niri/config.kdl` (mutable, changes immediate)

### When to Use Each

| Approach | Use when |
|----------|----------|
| Normal (in-store) | Config is generated by Nix, rarely changes, shared across machines |
| mkOutOfStoreSymlink | Hand-written config, frequently tweaked (keybinds, colors, editor) |

## Git + NixOS: Layers of Safety

NixOS gives you independent rollback mechanisms:

1. **Git history** — config source control
2. **NixOS generations** — bootable system snapshots (created on every `nixos-rebuild switch`)
3. **Btrfs snapshots** — filesystem-level snapshots (optional)

### Key Gotcha: Generations ≠ Git State

Booting into an old generation does NOT change your git state. The generation runs the old *built* system, but your source files remain at the current commit. Running `rebuild` from an old generation builds from the *current* source, not the generation's.

| Problem | Solution |
|---------|----------|
| Config broke | `git checkout <good-commit>` + rebuild |
| System won't boot | Boot menu → select old generation |
| Deleted files | Btrfs snapshot restore |
| Emergency recovery | Bootloader → old generation |

**Best practice**: commit after every successful rebuild to keep git history synchronized with generations.
